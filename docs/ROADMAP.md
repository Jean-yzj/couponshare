# CouponShare 產品路線圖

撰於 2026-07-05,以 YC CEO review 模式產出(真實漏斗數據 + 市場掃描 + 逐項決策)。
一句話主軸:**先閉合兩個斷掉的迴路(拿券的人變成放券的人、分享行為變成拉新管道),再包成 App,最後才談商業化。**

閱讀方式:每個 Phase 都有「進入條件」與「驗收指標」。指標數字是可調的第一版假設,但「要量什麼」不可省略——沒有數字的功能不准開工。

---

## 1. 現況數據(所有決策的依據,2026-07-05 實測)

| 指標 | 數值 | 判讀 |
|---|---|---|
| 註冊會員 | 8,025 | 一天 +5,568 的 Threads 爆紅(來源已驗證) |
| 上架過券的人 | ~395(4.9%) | **供給瓶頸:95% 用戶只拿不放** |
| 累計上架 / 送出 | 845 / 446 | 交易閉環健康:上架的券過半數真的送出去了 |
| 朋友推薦註冊 | 8 人 | **病毒迴路斷裂:邀請機制形同虛設** |
| 平台 | 手機 web 為主、Google 登入 69% | 行動優先是既定事實,不是選項 |

健康的部分(不用動):交易閉環、評價、檢舉自動下架(3 次)、帳號自動停權、管理後台、每日備份、反濫用限流。
斷裂的部分(路線圖主軸):供給轉換 4.9%、推薦 8 人、通知只有站內(用戶不回來就看不到)。

## 2. 市場位置(2026-07-05 掃描,來源存於審查記錄)

- 台灣現況:券轉讓分散在 Dcard 票券板(搶留言)、PTT(站內信議價)、LINE 群組(封閉靠人脈)、蝦皮(商業買賣+手續費)。共通痛點:**無結構、詐騙風險高、曝光靠緣分**。
- 國際:CardCash、swap.coupons、Raise 全是「禮品卡買賣/折現」生意;**「純免費贈送」定位查無存活先例**——這是空白機會,也是警示(沒人證明過它能活)。
- CouponShare 差異化:純贈送文化 + 結構化配對 + 無金流。無金流同時是護城河(繞開詐騙誘因與法規地雷)與紅線(見第 8 節)。

## 3. 原則(每個新功能開工前先過這三關)

1. **餵迴路才准做**:這功能讓「拿券→放券」或「分享→拉新」哪個迴路更順?都沒有就進 backlog。
2. **不碰金流、不做點數換價值**:貢獻分只換榮譽與額度,永不換錢。這是法規與詐騙的防火牆。
3. **克制**(focus over sprawl):同時進行的主打功能不超過一個 Phase;新功能上線兩週內必須回看指標,沒動靜就砍。

## 4. Phase 1 — 供給激活(現在~兩週,web)

時間註:「兩週」包含部署、觀察與微調;純開發工時約兩天(CC)。
(2026-07-05 場外審查後修訂:原 1C 許願牆經使用者否決移除——**不做任何索取型功能**,記錄於第 8 節。)

**前置作業(動工前完成,缺一不開工):**
1. 產出「週完成交易數」基線報表(CC 半天)——第 9 節的北極星目前只有累計數 446,沒有週基線,Phase 3/4 的門檻就無從定義。
2. 完成 R2 離站備份設定(唯一等待使用者手動的步驟;本機每日備份已在跑,但同機備份擋不了磁碟故障)。

### 1A. 首次分享引導
- **做什麼**:新用戶註冊後(或第一次領券成功後)出現一次性引導卡:「把一張你用不到的券放上來」,搭配簡化版三步上架流程(拍照/截圖 → 自動帶入 → 完成),完成後給「初次分享」徽章 + 貢獻分。
- **為什麼**:4.9% 是全產品最貴的數字。領券成功的當下是情感高點(剛收到別人的善意),是轉化供給者的最佳時機。
- **邊界情況**:略過引導的人一週後在 feed 頂部再溫柔提示一次,之後永久安靜;已上架過的人永不顯示。
- **驗收指標**(場外審查 #2/#8 修訂:主指標改新用戶 cohort,不背爆紅日沉睡者的債):(1) 主指標 = 新註冊者 7 日內首次上架率 + 每週首次上架人數(先取兩週基線再定目標);(2) 引導漏斗 = 看到引導卡 → 完成上架 轉換率(埋點已建);(3) 結果綁定 = 經引導上架的券 7 日內收到申請的比例(防「為指標而上架」的虛胖);(4) 全站分享率 4.9%(2026-07-05)僅作長期參考,不當驗收門檻。
- **工作量**:CC 約半天。

### 1B. 券連結歸因 + 動態分享卡
- **做什麼**:(a) 每張券頁支援 `?ref=<userId>`,未登入者領券前需註冊,註冊時把 ref 寫進 `referredById`(沿用既有欄位與 +2 額度獎勵);(b) 券詳情頁的分享按鈕產生帶 ref 的短連結;(c) 每張券動態生成 OG 分享圖(品牌色卡片:券名、品牌、到期日),讓連結貼到 LINE/Threads 時是一張漂亮卡片而不是純文字。
- **為什麼**:推薦僅 8 人的根因不是獎勵太小,是「邀請」和用戶的自然行為脫節。用戶本來就會把券連結傳給朋友——把這個既有行為變成歸因管道,分享券=邀請,病毒迴路才會轉。爆紅來自 Threads 也證明:這個產品天生靠「一張券的連結」傳播。
- **邊界情況**:自己 ref 自己不算;已註冊者點 ref 連結直接看券不受影響;ref 無效(用戶已刪除)時靜默忽略。
- **驗收指標**(場外審查 #3/#8 修訂):先埋三段漏斗(點分享 → 外部開啟 → 完成註冊),兩週基線後才定正式目標(暫定伸展值每週 100 人,承認是假設);結果綁定 = ref 註冊者 7 日內完成第一筆領券的比例。
- **工作量**:CC 約一天(OG 圖用 Next.js ImageResponse,零新依賴)。

### Phase 1 失敗模式與防護

| 失敗模式 | 症狀 | 防護 |
|---|---|---|
| 引導變騷擾 | 略過後一直跳 | 一次性 + 一週後僅一次補提示,永久記錄於用戶偏好 |
| ref 濫用刷額度 | 自導自演註冊 | 沿用既有邀請防護(同裝置/IP 節流);+2 額度本身價值低,誘因有限 |
| OG 圖洩漏隱私 | 分享卡帶到個資 | 卡片只含券公開欄位(名稱/品牌/到期日),絕不含條碼與擁有者資訊 |

**Phase 1 驗收方式(場外審查 #6 修訂:兩個實驗各自獨立判定,互不拖累,也不 gate App 軌道):** 1A 與 1B 上線兩週取得基線 → 定正式目標 → 四週檢討:達標留、未達修或砍。Phase 2(App)不等這裡的結果,見下節。

## 5. Phase 2 — 直接做 App Store 原生 App(2026-07-05 使用者定案)

使用者裁決:**跳過 PWA/TWA,直上 iOS 原生 App**(推翻 APP-PLAN.md 的 Stage 0/1 漸進路線;該文件的 Stage 2 章節與 docs/IOS-APP-SPEC.md 仍是實作依據)。

- **為什麼現在可行**:後端 App 端點其實已備——`auth/token`(Bearer 登入)、`auth/apple`、`auth/google/native`、`me/push-tokens`、`me/blocks` 都已存在,APP-PLAN 撰寫時列的「缺的東西」多數已補完。剩餘合規缺口:App 內刪除帳號端點與設定頁入口(App Store 5.1.1(v) 必查)。
- **時機**:Phase 1 功能上線後即啟動,與觀察期並行,不等 Phase 1 指標(場外審查 #1 的排序問題以此解決)。
- **通知閉環**:由原生推播(APNs)承擔——「有人申請你的券你卻不知道」的正解。LINE 官方帳號免費額度僅 200 則/月(硬上限),不可行。
- **已知代價(誠實記錄)**:(1) App 上線前 web 用戶的通知缺口持續存在;(2) iOS 不覆蓋 Android(台灣約半數用戶),Android 之後以 TWA 打包補上(2-3 天,列 backlog);(3) 這是全路線圖最大單項投資(以週計),期間 web 端只維運不加新功能——單人帶寬的取捨,寫明避免中途自我懷疑。
- **送審合規**:照 APP-PLAN 審查清單(Apple 登入端點已有、封鎖已有、刪帳號要補、隱私營養標籤對照隱私權政策填)。

## 6. Phase 3 — 流動性深化(1-3 個月,條件式候選)

進入條件:Phase 1 實驗有結論、iOS App 上線。候選(屆時再排序,不預先承諾):
- 到期救援推播化(48 小時到期券推給追蹤該品牌的人)
- 搜尋與瀏覽升級(品牌自動完成、全文搜尋)
- 信任徽章(edu 信箱驗證「學生認證」——註冊數據顯示學生族群密集)
- cohort 留存報表(做 iOS 決策的前置需求,可提前)

## 7. Phase 4 — 商業化實驗(3-12 個月,條件式)

- **啟動條件**:週完成交易數穩定(基線見第 9 節)+ 供給不再是瓶頸(分享率 ≥ 15%)。
- **形式**:品牌官方專區——品牌帳號認證 + 官方券投放 + 品牌數據面板(自家券的領取/追蹤數)。對品牌的賣點是既有的品牌追蹤與正規化資料。
- **紅線**:不做抽成(無金流可抽)、不做蓋版廣告、官方券必須明確標示。變現破壞贈送文化是這類平台已知死因,任何商業功能上線後監測檢舉率與留存,異常即回退。

## 8. 暫不做(backlog,附理由——2026-07-05 使用者裁決)

| 項目 | 理由 | 撿回條件 |
|---|---|---|
| 許願牆/任何索取型功能 | 使用者否決(2026-07-05;場外審查 #4/#5 同方向):多開一個 UGC 審核面、且可能助長索取文化 | 無預設條件——除非使用者主動重啟 |
| PWA / TWA | 使用者定案跳過,直上 iOS;PWA 的推播價值由 APNs 取代 | iOS 站穩後以 TWA 補 Android 覆蓋(工時 2-3 天) |
| 內容 AI 預審 | 3 檢舉自動下架已擋大部分;重心轉 App | 惡意上架頻率 > 每週 5 件時 |
| LINE 官方帳號通知 | 免費額度 200 則/月硬上限,規模不符;App 推播將取代 | 願意付月費且推播開啟率不足時 |
| 門市/地點維度 | 增加上架摩擦,與「3 秒上架」方向矛盾;且目前無地區資料 | 用戶檢舉「找不到適用門市」成為主要客訴時 |
| email digest | 通知主力押 Web Push;email 開信率低 | 推播被證明觸達不足時 |
| 成就/連續分享 streak | 已有等級+貢獻分+排行榜,再疊會稀釋 | 現有激勵被證明無感時 |
| 多語/國際化 | 台灣市場都還沒吃透 | 不設條件,一年內不碰 |

## 9. 指標框架

- **北極星:週完成交易數**(一張券真的從 A 手上到 B 手上才算數;上架數、註冊數都只是輸入)。
- 目前只有累計數(446),**第一件事是量出週基線**,才能定 Phase 3/4 的門檻數字。
- 漏斗儀表已有(admin 頁 activation funnel):註冊 → 分享 → 領券 → 完成。
- 補上:cohort 留存(D1/D7/D30)、上架→首次申請中位時間(流動性體感)、guardrail 指標(檢舉率、爭議率——成長不能以信任為代價)。
- **KPI 紀律(場外審查 #8)**:任何過程指標(註冊、分享、點擊)都必須綁一個交易結果指標(申請率、完成率、完成時間)一起看,防止優化到虛胖。

## 10. 風險登記簿(對照 C2C 平台已知死因)

| 死因 | 本產品現況 | 防護 |
|---|---|---|
| 供給枯竭 | 4.9% 分享率,最大風險 | Phase 1 全力攻擊;週上架數進儀表盯 |
| 詐騙侵蝕信任 | 條碼加密、評價、3 檢舉下架、爭議流程已在 | 監測檢舉率;預審留在 backlog 隨時可撿 |
| 區域供需失衡 | 無地區資料,現況未知 | 先不收地區資料(降摩擦優先);若客訴浮現再議 |
| 法規(券被視為有價證券) | 免費贈送 + 無金流 = 低風險 | 紅線:永不引入金流與點數變現 |
| 品牌方異議(場外審查 #7;比證券法更現實) | 多數券條款寫「不得轉讓」;條碼截圖可能含個資;品牌可能發函要求下架 | 待辦:盤點前 20 品牌券條款;補「品牌方申訴」快速下架通道(檢舉機制已可承接);分享卡永不含條碼 |
| 變現破壞社群氛圍 | 尚未變現 | Phase 4 紅線條款 + 異常回退機制 |
| 單人維運 | 每日備份已建、自動下架/停權已自動化 | R2 離站設定已列 Phase 1 前置作業第 2 項;管理負擔重的功能一律先自動化再上線 |
